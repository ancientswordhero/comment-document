# 评论回复提示 & 完整对话视图 设计文档

**日期：** 2026-06-04  
**状态：** 已确认

---

## 概述

两项前端改进，提升评论区的回复体验：
1. 回复时显示「回复 @用户名」标签，明确回复对象
2. 当某条二层回复下的子对话达到一定规模时，支持弹窗查看完整对话时间线

后端需小幅改动：将回复嵌套逻辑从「仅根评论」扩展为「递归填充」，使子回复也能携带其下的回复树。

---

## 一、回复提示标签

### 位置

`ReviewItem.vue` 中回复输入区（`showReplyInput` 为 true 时）。

### 行为

点击某条评论的「回复」按钮后，输入区顶部新增标签：

```
回复 @张三  [×]
┌─────────────────────┐
│ 写下你的回复...       │
└─────────────────────┘
          [取消] [回复]
```

- 标签显示被回复者用户名
- 点击用户名跳转到该用户主页（emit `view-user`）
- 点击 × 或「取消」关闭输入区并清空内容
- 标签和按钮颜色使用主题色 `--color-primary: #c9a96e`

### 实现

- `ReviewItem.vue` 已有 `review.username`，直接从 prop 取
- 回复标签仅在 `isReply=true`（子回复）时显示回复对象的用户名
- 根评论的回复输入框显示「回复 @根评论用户」

## 二、完整对话视图

### 对话线程定义

```
根评论（用户1）
  ├─ 回复A（用户2→用户1）  ← 对话线程的"起点"（parent_id = 根评论ID）
  │    ├─ 回复A1（用户1→用户2）
  │    ├─ 回复A2（用户3→用户2）
  │    ├─ 回复A3、A4、A5   ← 累计≥5 → 触发
  │
  ├─ 回复B（用户3→用户1）  ← 另一条独立对话线程
  │    └─ 回复B1
```

- 每条**直接回复根评论**的回复自成一个「对话线程」
- 线程包含：根评论 + 该二层回复 + 其所有子孙回复
- 当线程消息总数 ≥ 6（即二层回复 + 其子孙 ≥ 5）时，显示入口

### 触发条件计算

在 `ReviewItem.vue` 中递归统计每个二层回复的子孙节点数：

```
1（根评论） + 1（二层回复） + 子孙回复数 ≥ 6
```

即：子孙回复数 ≥ 4 时触发。

### 入口 UI

符合条件的二层回复下方显示：

```
查看完整对话（N条）→
```

- N = 线程总消息数（根评论 + 二层回复 + 所有子孙回复）
- 颜色 `--color-primary: #c9a96e`
- 点击弹起 `ConversationModal`

## 三、对话弹窗 ConversationModal

### 新文件

`reader-app/src/components/ConversationModal.vue`

### Props

| Prop | 类型 | 说明 |
|---|---|---|
| visible | Boolean | 是否显示 |
| rootReview | Object | 根评论数据 |
| messages | Array | 按时间旧→新排序的消息列表（根评论 + 线程所有回复） |
| bookId | Number | 所属书籍 |
| currentUserId | Number | 当前用户ID |

### Events

| Event | 参数 | 说明 |
|---|---|---|
| close | - | 关闭弹窗 |
| reply | (parentId, content) | 发送回复 |
| like | (reviewId) | 点赞 |
| delete | (reviewId) | 删除 |
| edit | (reviewId, content) | 编辑 |
| report | (reviewId) | 举报 |
| view-user | (userId) | 查看用户主页 |

### 布局

```
┌───────────────────────────────────────┐
│  完整对话                         [×]  │
├───────────────────────────────────────┤
│  张三 · 2小时前                       │
│  这本书真的很好看，推荐给大家           │
│  ─────────────────────────────        │
│  李四 · 1小时前                       │
│  回复 @张三：同意！文笔也很好           │
│  张三 · 45分钟前                      │
│  回复 @李四：你最喜欢哪个章节？         │
│  ... 更多消息                         │
├───────────────────────────────────────┤
│  ┌───────────────────────────────────┐│
│  │ 写下你的回复...                    ││
│  └───────────────────────────────────┘│
│                    [取消] [发送回复]   │
└───────────────────────────────────────┘
```

- 宽度约 600px，最大高度 80vh，内容区可滚动
- 打开时自动滚到底部（最新消息）
- 根评论与后续回复用分隔线区隔
- 回复他人的消息在内容前带「回复 @用户名：」前缀
- 根评论不显示「回复 @用户名」前缀（它是对话的起点）
- 底部输入区固定，回复对象默认为当前线程的二层回复作者
- 用户自己的消息显示「(我)」标记
- 弹窗内的回复提交后实时追加到消息列表末尾

### 已编辑标记

消息若 `updatedAt !== createdAt`，显示「(已编辑)」标记（复用 ReviewItem 逻辑）。

## 四、数据流

### 后端改动

`ReviewService.getReviews()` 中，将回复分组方式从「按 root_id」改为「按 parent_id」：

- 现有：`repliesMap` key 为 root_id，仅根评论查找子回复
- 改为：`childrenMap` key 为 parent_id，任意评论都能找到其直接子回复
- `toResponse()` 移除 `parentId == null` 限制，递归填充所有层级

```
GET /api/books/{bookId}/reviews
  → 根评论（含完整 replies 树，每层都有嵌套）
  → 二层回复的 replies 不再为空，包含其所有子孙回复
```

改动范围：`ReviewService.java` 约 5 行。

### 前端数据流

```
ReviewSection.vue
  ├─ 获取评论列表 → rootReview.replies[].replies[] 完整树
  │
  ├─ ReviewItem.vue（顶层）
  │   ├─ 回复标签：显示 `回复 @username`
  │   ├─ 计算每个二层回复的线程消息数（递归统计其 replies）
  │   ├─ 显示「查看完整对话」入口
  │   └─ $emit 打开弹窗（带上 rootReview + 线程数据）
  │
  └─ ConversationModal.vue（弹窗）
      ├─ 展平线程树 → 按时间排序的消息列表
      ├─ 回复 → emit 到 ReviewSection → POST /api/reviews/{id}/reply → 刷新
      └─ 点赞/删除/举报 → emit 到 ReviewSection 处理
```

### 消息排序

弹窗内所有消息按 `createdAt` 升序排列（旧→新）。在调用 `createReply` 后刷新列表，确保新回复出现在最底部。

## 五、边界情况

| 场景 | 处理 |
|---|---|
| 线程消息恰好 5 条 | 不显示入口（阈值 ≥6） |
| 线程消息 ≥6 条 | 显示「查看完整对话」入口 |
| 弹窗中删除某条消息 | 删除后刷新列表，若消息数降回 <6，关闭弹窗并隐藏入口 |
| 用户在弹窗内回复 | 追加到消息列表末尾，计数器更新 |
| 未登录用户 | 弹窗仍可查看，但底部输入区显示登录提示 |
| 弹窗在手机端宽度 | 使用 95vw 宽度适配小屏 |
